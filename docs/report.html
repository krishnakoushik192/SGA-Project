Student-Course App: Design and Implementation Approach

1. Overview

This project is a Spring Boot MVC application that manages two related entities: Student and Course. The implementation follows a layered architecture:

  • Controller layer for request handling and JSP view binding
  • Service layer for business flow and update logic
  • Repository layer for persistence and custom join queries
  • JPA entities for schema design and validation constraints
  • JSP views for list, create, and update screens

The application currently supports:

  • List students
  • Create student
  • Update student
  • List students with joined course details
  • List courses
  • Create course
  • Update course

Design Rationale

The relationship is modeled as one Course -> many Students.

  • Course owns course-level data: title, unique code, and credits.
  • Student owns student-level data: name, unique email, and enrollment date.
  • course_id is a mandatory foreign key, which guarantees every student belongs to a valid course.
  • The unique email and course code constraints prevent duplicate logical records.
JPA Mapping

Course defines the one-to-many side:

@Entity
public class Course {
  @Id
  @GeneratedValue(strategy = GenerationType.IDENTITY)
  private Long id;

  @NotBlank(message = "Course code is required")
  @Column(nullable = false, unique = true, length = 20)
  private String code;

  @OneToMany(mappedBy = "course", cascade = CascadeType.ALL, orphanRemoval = false)
  private List<Student> students = new ArrayList<>();
}

Student defines the many-to-one side:

@Entity
public class Student {
  @NotBlank(message = "Email is required")
  @Email(message = "Email must be valid")
  @Column(nullable = false, unique = true)
  private String email;

  @NotNull(message = "Course is required")
  @ManyToOne(fetch = FetchType.LAZY)
  @JoinColumn(name = "course_id", nullable = false)
  private Course course;
}

This is a good fit because the actual foreign key is stored on the Student table. It keeps the database relationship simple and makes it easy to list students with their assigned courses.

3. Implementation Details for Each Operation

3.1 Student Operations

A. List Students with Course Details

Flow
  1. GET /students reaches StudentController.listStudents.
  2. The controller calls studentService.findAllStudents().
  3. The controller also calls studentService.findStudentCourseDetails().
  4. The repository executes a custom JPQL inner join query.
  5. Data is passed to students/list.jsp.
Controller code
@GetMapping
public String listStudents(Model model) {
  model.addAttribute("students", studentService.findAllStudents());
  model.addAttribute("studentCourseDetails", studentService.findStudentCourseDetails());
  return "students/list";
}
Repository inner join query
@Query("""
  select new com.example.app.dto.StudentCourseView(
    s.id, s.name, s.email, s.enrollmentDate,
    c.id, c.title, c.code, c.credits
  )
  from Student s
  inner join s.course c
  order by s.name
  """)
List<StudentCourseView> findStudentCourseDetails();

This query returns only the fields needed for the joined table instead of exposing full entity graphs to the JSP page. The result is clearer and directly satisfies the requirement to perform an inner join between both entities.

B. Create Student

Flow
  1. GET /students/new returns the empty student form.
  2. The user enters student details and selects a course.
  3. POST /students binds fields to a Student object.
  4. Controller validates the submitted fields and selected course.
  5. Service resolves courseId to a managed Course.
  6. The student is saved with a valid foreign key relationship.
  7. Duplicate email conflicts are shown as form errors.
Controller-side binding
@PostMapping
public String createStudent(
  @Valid @ModelAttribute("student") Student student,
  BindingResult bindingResult,
  @RequestParam(value = "course.id", required = false) Long courseId,
  Model model,
  RedirectAttributes redirectAttributes
) {
  validateCourseSelection(courseId, bindingResult);
  if (bindingResult.hasErrors()) {
    addStudentFormAttributes(model, "Add Student", "/students");
    return "students/form";
  }

  try {
    studentService.createStudent(student, courseId);
    return "redirect:/students";
  } catch (DataIntegrityViolationException ex) {
    bindingResult.rejectValue("email", "duplicate", "Student email must be unique.");
    return "students/form";
  }
}
Service logic
public Student createStudent(Student student, Long courseId) {
  student.setCourse(findCourse(courseId));
  return studentRepository.save(student);
}

C. Update Student

Flow
  1. GET /students/{id}/edit loads the student by id.
  2. Existing values are pre-filled in the JSP form.
  3. POST /students/{id} validates submitted fields.
  4. Service loads the existing persistent entity.
  5. Only intended fields are changed: name, email, enrollment date, and course.
  6. Duplicate email conflicts are caught and shown to the user.
Update service code
public Student updateStudent(Long id, Student updatedStudent, Long courseId) {
  Student student = findStudent(id);
  student.setName(updatedStudent.getName());
  student.setEmail(updatedStudent.getEmail());
  student.setEnrollmentDate(updatedStudent.getEnrollmentDate());
  student.setCourse(findCourse(courseId));
  return studentRepository.save(student);
}

This approach first loads the existing entity and then mutates only the allowed fields. It avoids replacing the whole database row with unchecked submitted form data.

Flow
  1. GET /courses/new returns the empty course form.
  2. The user enters code, title, and credits.
  3. POST /courses validates the submitted course.
  4. The service saves the course through the repository.

@PostMapping
public String createCourse(
  @Valid @ModelAttribute("course") Course course,
  BindingResult bindingResult,
  Model model,
  RedirectAttributes redirectAttributes
) {
  if (bindingResult.hasErrors()) {
    model.addAttribute("formTitle", "Add Course");
    return "courses/form";
  }

  try {
    courseService.createCourse(course);
    return "redirect:/courses";
  } catch (DataIntegrityViolationException ex) {
    bindingResult.rejectValue("code", "duplicate", "Course code must be unique.");
    return "courses/form";
  }
}

Database Choice

H2 in-memory database was used because it gives:

  • Zero external database setup
  • Fast startup for demos and testing
  • Deterministic clean state with Hibernate create-drop

View Technology

JSP was used with Spring form tags because:

  • It matches the Spring MVC assignment requirement.
  • It supports server-rendered forms and validation messages.
  • It keeps the stack simple for a CRUD-style application.

Validation and Error Handling

Validation is defined at entity level using Jakarta Validation annotations:

  • @NotBlank for required text fields
  • @Email for student email format
  • @NotNull for enrollment date, course, and credits
  • @Min and @Max for course credit range

Uniqueness conflicts are handled by catching DataIntegrityViolationException inside controllers and converting them into user-friendly form errors.

5. Challenges Faced and How They Were Overcome

Challenge 1: Modeling the student-course relationship cleanly in forms

Problem

The UI needs the user to choose a course while creating or updating a student, but HTML forms naturally submit simple values rather than full entity objects.

Approach

The form submits course.id, and the service resolves it into a managed entity:

student.setCourse(findCourse(courseId));

Why this solved it

  • It avoids brittle nested object binding.
  • It keeps persistence logic inside the service layer.
  • It guarantees the student is assigned to an existing course.

Challenge 2: Showing joined student and course data efficiently

Problem

The student list needs both student fields and course fields in one table.

Approach

I used a repository-level JPQL inner join with a record projection:

List<StudentCourseView> findStudentCourseDetails();

Why this solved it

  • It returns exactly the fields required by the screen.
  • It keeps the JSP simple.
  • It clearly demonstrates the required inner join.

Challenge 3: Preventing duplicate logical records

Problem

Two students should not share the same email, and two courses should not share the same course code.

Approach

  • Enforced uniqueness at the database level with unique = true.
  • Caught DataIntegrityViolationException in controllers.
  • Converted the exception into a readable validation error.

Challenge 4: Updating existing entities safely

Problem

Directly saving the submitted form object can overwrite fields carelessly and makes update behavior harder to control.

Approach

The service first loads the existing persistent entity, then updates only intended fields.

Why this solved it

  • It keeps update behavior explicit.
  • It avoids accidental replacement semantics.
  • It works well with JPA-managed entities.

Challenge 5: Running Maven on a machine without global Maven

Problem

The mvn command was not available globally in PowerShell.

Approach

A local Windows Maven launcher, mvnw.cmd, was added so the project can run with:

.\mvnw.cmd spring-boot:run

Outcome

The application can be started from the project directory without installing Maven globally.

7. Conclusion

The implementation is a complete Spring Boot MVC CRUD application with a properly modeled one-to-many relationship between Course and Student. It uses JPA annotations for schema mapping and validation, Spring Data repositories for persistence, services for relationship assignment and update logic, controllers for JSP form workflows, and a JPQL inner join query for the combined student-course listing.

The result is easy to understand, easy to test, and appropriate for a small relational web application assignment.
